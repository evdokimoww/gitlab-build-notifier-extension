# GitLab build stage notifier

Расширение для **Chromium** (Chrome, Edge, Brave и др.): следит за **открытыми вкладками** с merge request или pipeline на вашем **GitLab**, периодически опрашивает **REST API** и **один раз** сообщает, когда все джобы выбранного **stage** (по умолчанию `build`) завершились успехом или ошибкой.

## Возможности

- Системное **уведомление** браузера с иконкой (успех / ошибка).
- Опциональный **звуковой сигнал** (через offscreen-документ): **разные файлы** для успеха (`sounds/notify.wav`) и ошибки (`sounds/notify-fail.wav`) — при желании замените их своими WAV с теми же именами.
- Префикс **`[CI OK]`** / **`[CI FAIL]`** в заголовке вкладки.
- **Мигание favicon** (`notify-ok.png` / `notify-fail.png`), затем стабильная статусная иконка.
- **Whitelist** путей проектов в GitLab (если нужно ограничить проекты).

Данные учётной записи GitLab **не покидают** ваш браузер: токен хранится в `chrome.storage.local`, запросы идут напрямую с вашей машины к API GitLab.

## Требования

- Браузер на базе Chromium с поддержкой **Manifest V3**.
- Для API: **Personal Access Token** в GitLab с правом **`read_api`** (или шире, если принято у вас в организации).
- Доступ к вашему GitLab по HTTPS (при первом сохранении «чужого» базового URL браузер может запросить **доступ к сайту** — его нужно выдать).

## Установка из исходников

1. Клонируйте репозиторий (или скачайте архив и распакуйте).
2. Откройте `chrome://extensions` (в Edge: `edge://extensions`).
3. Включите **Режим разработчика**.
4. Нажмите **Загрузить распакованное расширение** и укажите папку с `manifest.json`.

После обновления кода нажмите **Обновить** на карточке расширения.

## Настройка

1. ПКМ по иконке расширения → **Параметры** (или ссылка «Настройки» в попапе).
2. **Базовый URL GitLab** — например `https://git.example.com` (без лишнего слэша в конце).
3. **Personal Access Token** — создайте в GitLab: *Preferences → Access Tokens* (или *Edit profile → Access Tokens*), вставьте токен в поле.
4. При необходимости измените:
   - **имя stage** (по умолчанию `build`);
   - **интервал опроса** (секунды, ограничен диапазоном в интерфейсе);
   - трактовку **skipped** / **canceled**;
   - включение **мигания favicon** и **звука**.

Кнопка **«Сбросить уже уведомлено»** очищает кэш: следующее завершение того же пайплайна снова вызовет уведомление (удобно для отладки).

## Как это работает

1. Service worker по таймеру смотрит **все открытые вкладки** и отбирает URL вида:
   - `…/-/merge_requests/<iid>…`
   - `…/-/pipelines/<id>…` (в том числе с хвостом пути: graph, failures и т.п.)
   - `…/-/jobs/<id>…` (страница лога джоба — пайплайн берётся из API по этому джобу)
2. Базовый URL вкладки должен совпадать с настроенным инстансом GitLab.
3. Для MR определяется пайплайн (`head_pipeline` или последний пайплайн MR).
4. Запрашиваются джобы пайплайна; фильтруются по имени **stage** (сравнение **без учёта регистра**, `build` и `Build` совпадают).
5. Если в stage есть джоб в статусе **`failed`**, событие считается **ошибкой сразу** (fail-fast): не ждём остальные джобы вроде `manual` / `pending`, чтобы всё равно прийти к уведомлению, красному favicon и звуку ошибки. Иначе пока есть джобы в «живых» статусах (`running`, `pending`, `canceling`, …) — уведомление не шлётся.
6. Когда исход stage определён (успех или ошибка; с учётом флагов для **skipped** / **canceled** и нормализации статусов в API), один раз отправляются **уведомление**, опционально **звук** (успех или ошибка), обновляются **заголовок вкладки** и **favicon**; пара `(проект, pipeline, stage)` сохраняется, чтобы не дублировать событие.

## Разрешения расширения

| Разрешение        | Зачем |
|-------------------|--------|
| `tabs`            | Найти открытые вкладки MR/pipeline и при необходимости выполнить скрипт на странице (favicon, запасной звук). |
| `storage`         | Настройки и кэш «уже уведомлено». |
| `notifications`   | Системные уведомления. |
| `alarms`          | Периодический опрос без постоянного таймера вручную. |
| `scripting`       | Вставка скрипта на вкладку GitLab (favicon, заголовок, запасной звук). |
| `offscreen`       | Фоновая страница для воспроизведения звука (service worker не может надёжно вызывать `Audio`). |
| host permissions  | Запросы к API GitLab (`/api/v4/…`) и работа на страницах инстанса. |

## Устранение неполадок

- **Нет всплывающего уведомления**  
  В настройках ОС разрешите уведомления для браузера. На macOS: *Системные настройки → Уведомления → [Chrome/Edge]*. Убедитесь, что для расширения не заблокированы уведомления.

- **Нет звука**  
  Проверьте, что в настройках расширения включён звук. Убедитесь, что после добавления разрешения `offscreen` расширение **перезагружено**. При блокировке автовоспроизведения может сработать запасной вариант через вкладку GitLab.

- **Событие не отслеживается**  
  Проверьте токен, базовый URL, whitelist проекта и что вкладка действительно на нужном хосте и в нужном пути MR/pipeline.

## Структура репозитория

| Файл / папка        | Назначение |
|---------------------|------------|
| `manifest.json`     | Manifest V3, разрешения, фоновый скрипт. |
| `background.js`     | Опрос, логика stage, уведомления, вызов звука и скриптов на вкладке. |
| `gitlab-api.js`     | Обёртки над GitLab REST API v4. |
| `options.html/js`   | Страница настроек. |
| `popup.html/js`     | Краткий попап и ссылка на настройки. |
| `offscreen.html/js` | Воспроизведение звука по URL из сообщения (успех / ошибка). |
| `sounds/`           | `notify.wav` — успех; `notify-fail.wav` — ошибка (можно заменить своими). |
| `icons/`            | `extension-*.png` — иконка расширения на основе GitLab tanuki; **`notify-ok.png`** / **`notify-fail.png`** — иконки уведомлений (RGBA). |
